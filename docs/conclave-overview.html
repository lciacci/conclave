3× vLLM co-resident on one L40S · 48 GB
  • wireOpenAI-compatible everywhere. vLLM serves it, LiteLLM routes it, the client speaks it — and it touches zero OpenAI servers. Compatibility buys drop-in clients, not a dependency.
  • fleetLineage-decorrelated specialists. Qwen coder, a Qwen-distill reasoner, and Google's Gemma — chosen so their failure modes don't correlate. A judge feeds on disagreement.
  • memoryCumulative VRAM ceilings. vLLM's gpu_memory_utilization is a ceiling on total GPU memory, not a private slice. Models start one at a time with rising ceilings (0.25 → 0.55 → 0.82) so no late starter starves the KV cache.
  • accessTailscale-only, no exceptions. The inference box has no public ingress, ever. Reachable like a LAN device over WireGuard.

The thesis

The judge is the whole point

Arbiter, judge, mediator, critic, router — all meta-reasoners over specialized outputs. Naming that family is what turns two similar components into a design-pattern exploration. Conclave's judge runs per request.

  • pluggableThe judge target is a config value — model name + URL — not hardcoded. Swapping in-fleet ↔ frontier is a one-line change.
  • defaultIn-fleet Gemma-9B. Its lineage is decorrelated from the two Qwen candidates, so it's less likely to share their blind spots. No frontier model in the critical path.
  • evalFrontier is a baseline to beat, never the default. The identical ensemble was run with judge=gemma vs judge=frontier and graded blind against gold references. That comparison is the experiment — result below.
  • modesSelect or synthesize. Return the single best answer verbatim, or merge the correct points and drop the errors. Candidates + rationale ride in response metadata for debugging.

The open question · v3 baseline

Does one GPU timeshare three models — or serialize them?

Three vLLM processes on one L40S contend for the same compute units. The co-residency tax decides whether spending 4× on multi-GPU is justified. Measured 2026-07-10: +30% — real, but far from serialized. Compare the outcome against the hypothetical it was weighed against.

Build log

Controls first, compute second

v0.5
verified

Cost layer, zero GPU spend

Tiered budget alerts, a hard-stop Lambda that stops tagged instances, and an idle-stop alarm — all live and tested against a throwaway box before the first GPU ever launched. One forgotten weekend on a g6e ≈ the whole monthly cap.

v1
verified

A single 70B-class model, serving tokens

Qwen2.5-72B AWQ (~40 GB, 4-bit) on one L40S via vLLM, reachable over Tailscale. Hit KV-cache starvation on first serve — fixed with --enforce-eager and a tuned memory ceiling. Torn down to $0.

v2
verified

Gateway + a 3-model fleet

LiteLLM in front of three co-resident specialists on one GPU, each serving clean through the gateway, with per-model cost accounting and a GPU-utilization idle-stop metric. The 32B coder didn't fit three-way — dropped to 14B; it returns on its own GPU later.

v3
thesis proven

Ensemble + judge — the thesis

Client-side orchestrator built and verified. Contention baseline captured (+30% → multi-GPU dismissed); the fleet boots clean on additive memory slices. Judge eval run: the in-fleet Gemma-9B judge ties a frontier judge on 15 of 18 queries — a small self-hosted judge holds up. The pedagogically interesting phase, done.
